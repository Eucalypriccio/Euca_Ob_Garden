## EdgeLLM: Fast On-Device LLM Inference With Speculative Decoding

### 投机解码 speculative decoding

- 将大部分token的生成任务分派给小型的、常驻内存的 draft LLM
- 一个大型的target LLM作为验证者，它以自己的输出作为标准答案，对比draft LLM生成的token，如果有错误就纠正
- 打破内存墙-小型模型参数少，可以加载到快速内存（GPU的显存/CPU的缓存），避免了从慢速DRAM加载大模型的瓶颈
- 更好的扩展性-小模型参数少，更容易部署和扩展到不同的边缘设备
- 引入token树-draft LLM在每一步推理时生成多个可能的token，每一个token作为一个结点，形成了树状结构

### 具体设计

#### Compute-efficient branch navigation and verification

##### Navigation

- Confidence-based branch pacer：解决的是选择哪个分支（哪个分支应该得到更多的资源）的问题
- Branch decoder-解决的是如何在不同分支之间高效地生成token
	- 采用掩码
	- 生成后续的token，需要的是这个分支之前所有token的信息
	- 具体实现，我觉得是使用了树层次遍历的思想，把每一层的输出拼接在一起，形成**一个**线性序列，要生成下一个token，就用掩码把其他分支的token掩盖掉
	- 这样就不需要在不同的分支之间进行切换，只需要动态地修改这个掩码表

##### Verification

- Batched non-autoregressive verification
	- 把所有分支分批次，TLLM一次处理一批
	- 验证和加载参数可以同时进行
- Verified sequences merging
	- 验证过后，把每一层的相同结点合并为一个结点，最终一定是一个序列
	- 如果这个序列中包含修正的token，送回到发生错误的位置

#### Self-adaptive fallback strategy

- Tree-cumulative confidence-分支的累计可信度的最大值，来反映整个树的可信度
- Self-adaptive threshold
	- 允许DLLM保留其生成过程的最小累计可信度
	- 一旦 $T_c$ 低于这个阈值，就启动验证

#### Provisional generation pipeline

在TLLM进行验证的时候，DLLM继续生成token

## Ideas

>Despite these advantages, applying speculative decoding for on-device LLM still encounters inefficient issues. Preliminary experiments on Jetson TX2 and Xiaomi 10 using mT5 and LLaMA2 models show that per-token generation takes an average of 1.8 seconds and 6.1 seconds, respectively, even with speculative decoding.
